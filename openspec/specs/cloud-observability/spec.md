# cloud-observability Specification

## Purpose
OpenTelemetry traces/logs/metrics from backend, worker, and SPA into Azure Application Insights, plus alert rules and an operations dashboard.

## Requirements

### Requirement: The backend emits OpenTelemetry traces, logs, and metrics to Application Insights

The system SHALL provide `app/infrastructure/observability.py::init()` that initialises the OpenTelemetry SDK with the Azure Monitor OTLP exporter when `APPLICATIONINSIGHTS_CONNECTION_STRING` is set, and SHALL auto-instrument FastAPI, httpx, SQLAlchemy, and Celery. When the env var is absent, `init()` SHALL be a no-op.

#### Scenario: Backend traces appear in Application Insights

- **WHEN** the backend is running with `APPLICATIONINSIGHTS_CONNECTION_STRING` set
- **AND** an authenticated client makes a `GET /api/v1/components/lookup?mpn=NE555P` request
- **THEN** a `requests` entry appears in Application Insights with operation_name `GET /api/v1/components/lookup`
- **AND** the entry includes child spans for each outbound httpx call to the supplier APIs
- **AND** the entry includes a child span for the Redis `GET` cache check

#### Scenario: Local development without the env var is a no-op

- **WHEN** the backend is started with `APPLICATIONINSIGHTS_CONNECTION_STRING` absent or empty
- **THEN** `observability.init()` returns immediately
- **AND** the backend boots normally
- **AND** no OpenTelemetry exporter is registered

### Requirement: structlog records carry the active trace_id and span_id

The system SHALL extend the structlog processor chain so every log record emitted while a trace is active includes the W3C `trace_id` and `span_id` of the current span. App Insights uses these to correlate logs with traces in the UI.

#### Scenario: A backend log emitted inside a request handler carries the trace id

- **WHEN** the backend handles `GET /api/v1/components/lookup?mpn=NE555P` and logs `"supplier_lookup.cache.hit"` via structlog
- **THEN** the JSON log record contains a `trace_id` field whose value matches the trace_id of the parent HTTP span
- **AND** App Insights renders the log inline with the corresponding request entry

### Requirement: The frontend emits page-view + RUM telemetry to the same Application Insights instance

The system SHALL provide `frontend/src/lib/telemetry.ts::init()` using `@microsoft/applicationinsights-web` with the connection string from `import.meta.env.VITE_APP_INSIGHTS_CONNECTION_STRING`. The init MUST be called from `frontend/src/main.tsx` before the React tree renders. When the env var is absent, `init()` SHALL be a no-op.

#### Scenario: Page views appear in Application Insights

- **WHEN** a user opens `https://ada.tierra.audio/`
- **AND** the SPA boots with `VITE_APP_INSIGHTS_CONNECTION_STRING` injected at build time
- **THEN** a `pageViews` entry appears in Application Insights with `name="ada-asm/dashboard"` (or whatever route is initially loaded)

#### Scenario: SPA build without the env var is a no-op

- **WHEN** the SPA is built with `VITE_APP_INSIGHTS_CONNECTION_STRING` empty
- **THEN** `telemetry.init()` does not register the Web SDK
- **AND** no network requests are made to the Azure Monitor ingestion endpoint

### Requirement: Cross-tier traces are correlated via `traceparent`

The system SHALL propagate the W3C `traceparent` header from the frontend to the backend on every API call so a single end-to-end trace is visible in App Insights from page-view through SQL query.

#### Scenario: traceparent flows from the FE through to the BE

- **WHEN** the frontend makes a `GET /api/v1/components` call
- **THEN** the request includes a `traceparent` header generated by the Web SDK
- **AND** the backend's OpenTelemetry instrumentation extracts that header
- **AND** the backend's request span lists the FE's span as its parent
- **AND** App Insights "End-to-end transaction" view shows both spans linked

### Requirement: Sampling preserves all HTTP spans, samples SQL spans at 10%

The system SHALL configure the backend's OpenTelemetry sampler to keep 100% of HTTP request spans (including the parent of every trace) and 10% of SQL spans in prod. Dev SHALL keep 100% of both.

#### Scenario: HTTP spans are never dropped

- **WHEN** 100 HTTP requests hit the backend
- **THEN** 100 `requests` entries appear in Application Insights

#### Scenario: SQL spans are sampled in prod

- **WHEN** 100 HTTP requests each trigger 5 SQL queries on the prod backend
- **THEN** Application Insights shows approximately 50 SQL `dependencies` entries (10% of 500), not 500
